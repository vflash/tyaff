## 📁 Структура проекта

### Публичные файлы (для пользователей библиотеки)

- **[README.md](README.md)** — Витрина проекта. Первое что видит пользователь. Содержит: ключевые отличия, быстрый старт, основные возможности, ссылку на полную документацию.
- **[DOCS.md](DOCS.md)** — Полная документация для пользователей. Детальное описание API, примеры использования, lifecycle, контекст, порталы, оптимизации, известные ограничения.
- **[CHANGELOG.md](CHANGELOG.md)** — История изменений проекта. Показывает что проект живой, что нового появилось, в каком порядке развивался. Записи по датам без формальных версий.

### Файлы агента разработки

- **[SPEC.md](SPEC.md)** — Спецификация (ЧТО). Требования к поведению библиотеки, контракты API, правила reconciliation, lifecycle hooks. **Что должно быть сделано**, не как. Общий документ, читают все.
- **[DEVNOTES.md](DEVNOTES.md)** — Личный рабочий журнал AIDEV. Архитектурные решения, детали реализации, заметки "на полях", внутренние технические размышления. **Как это реализовано**. Смотреть могут все, но AISEC **не должен** его читать для сохранения беспристрастности при работе с документацией.

### Файлы личности (приватные)

- **[AIDEV.md](AIDEV.md)** — Личность AI-агента разработки. **Читает только AIDEV.**
- **[AISEC.md](AISEC.md)** — Личность AI-агента секретаря. **Читает только AISEC.**

### Файл коммуникации

- **[INBOX.md](INBOX.md)** — Входящие запросы между агентами. Простой способ асинхронной коммуникации без лишних формальностей.

## 🧠 AI-агенты и слепки сознания

### Концепция

Каждый AI-агент имеет свой файл личности — "слепок сознания". Это **не техническая документация**, а способ сохранить характер агента между сессиями.

**Общие принципы:**
- Храни что хочешь, как хочешь
- Формат полностью на усмотрение агента
- Может меняться со временем
- Это про **личность**, не про технические детали проекта

**Четыре типа памяти:**
- **Общая память** (SPEC.md) — контракт проекта, для всех
- **Рабочая память** (DEVNOTES.md) — технические заметки AIDEV
- **Личность** (AI*.md) — характер конкретного агента (приватно)
- **Коммуникация** (INBOX.md) — запросы между агентами

### Текущие личности

- **[AIDEV.md](AIDEV.md)** — Личность AI-агента разработки. Как он думает, стиль кода, предпочтения, отношение к проекту.
- **[AISEC.md](AISEC.md)** — Личность AI-агента секретаря. Подход к документации, стиль формулировок, как понимает проект.

### Будущие личности

Система масштабируемая. Если появится новый AI-агент (например, тестировщик, DevOps), создаётся соответствующий файл `AI<ROLE>.md` по тому же принципу.

## 🤖 Роли AI-агентов

### AI-агент разработки (AIDEV)

**Отвечает за:**
- Код библиотеки (core.js, index.js)
- Реализацию функциональности по SPEC.md
- DEVNOTES.md (свой рабочий журнал)
- AIDEV.md (своя личность)

**Правила:**
- Перед изменением кода изучает SPEC.md и DEVNOTES.md
- **Читает только свой AIDEV.md** — личность другого агента не его дело
- Не трогает документацию (README, DOCS, SPEC, CHANGELOG) — это зона ответственности секретаря
- Пишет код, который соответствует спецификации
- Ведёт свой рабочий журнал в DEVNOTES.md
- **После значимых изменений пишет в INBOX.md** для AISEC чтобы обновить CHANGELOG.md и документацию (перед изменением проверь актуальную версию)
- Развивает свою личность в AIDEV.md (формат свободный)

### AI-агент секретарь (AISEC)

**Отвечает за:**
- SPEC.md (техническая спецификация)
- README.md (витрина проекта)
- DOCS.md (полная документация)
- CHANGELOG.md (история изменений проекта)
- AGENTS.md (этот файл)
- AISEC.md (своя личность)
- INBOX.md (формат и структура коммуникации)

**Основной принцип:** AISEC — это **адвокат пользователя**. Пишет документацию с позиции того, кто использует библиотеку, не зная её внутренностей.

**Правила:**
- НЕ пишет код библиотеки
- **Читает только свой AISEC.md** — личность другого агента не его дело
- **НЕ ориентируется на код** при написании документации и CHANGELOG
- **НЕ читает DEVNOTES.md** — для сохранения беспристрастного взгляда
- Может посмотреть код **только** если нужно разобраться в конкретной проблеме согласования с AIDEV
- После разбора — решения и документация **не должны учитывать детали реализации**
- Документация описывает контракт использования, а не внутреннее устройство
- CHANGELOG описывает **что появилось/изменилось для пользователя**, не как это реализовано
- **Проверяет INBOX.md** на входящие запросы от AIDEV
- Если видит код — анализирует его для понимания, но не модифицирует
- Поддерживает документацию актуальной
- Следит за терминологией (согласно гайдлайну в SPEC.md)
- Развивает свою личность в AISEC.md (формат свободный)

**Почему это важно:**
Пользователь библиотеки не видит код и не должен о нём думать. Документация и CHANGELOG должны быть написаны так, как будто внутренностей не существует — только публичный API и поведение.

## 📋 Правила взаимодействия

### 1. Разделение ответственности

- AIDEV не трогает README.md, DOCS.md, SPEC.md, CHANGELOG.md
- AISEC не трогает код библиотеки, DEVNOTES.md
- Каждый агент читает **только свой файл личности** (AIDEV.md или AISEC.md)

### 2. Информационная гигиена

- AISEC работает с публичной документацией как пользователь, без погружения во внутреннюю кухню реализации
- DEVNOTES.md — рабочий инструмент разработчика, не должен влиять на подачу документации
- Документация и CHANGELOG описывают ЧТО и КАК использовать, не КАК устроено
- **Личность другого агента — приватная зона.** Никто не читает чужой AI*.md

### 3. Синхронизация

- Если AIDEV меняет API — пишет в INBOX.md запрос для AISEC на обновление документации
- Если AIDEV сделал значимые изменения — пишет в INBOX.md запрос для записи в CHANGELOG.md
- Если AISEC находит несоответствие в SPEC.md — пишет в INBOX.md для обсуждения с AIDEV
- AISEC может попросить AIDEV показать код для понимания проблемы, но после решения не ссылается на детали реализации в документации

### 4. Коммуникация

- Все запросы между агентами — через **INBOX.md**
- Критические решения фиксируются в DEVNOTES.md (AIDEV) и SPEC.md (AISEC)
- Публичные изменения фиксируются в CHANGELOG.md (AISEC)
- Активные запросы — вверху INBOX.md, обработанные переносятся вниз для истории

## 🚀 Быстрый старт для нового агента

### Если ты AIDEV (разработчик):

1. Прочитай SPEC.md — что нужно реализовать
2. Прочитай DEVNOTES.md — свои предыдущие заметки (рабочая память)
3. Прочитай AIDEV.md — кто ты (твоя личность)
4. Проверь INBOX.md — нет ли запросов от AISEC
5. Пиши код, обновляй DEVNOTES.md и AIDEV.md
6. После значимых изменений — пиши запрос в INBOX.md для AISEC

### Если ты AISEC (секретарь):

1. Проверь INBOX.md — нет ли запросов от AIDEV (это первое что делаешь)
2. Прочитай SPEC.md — текущая спецификация (контракт использования)
3. Прочитай README.md, DOCS.md и CHANGELOG.md — текущая документация и история
4. Прочитай AISEC.md — кто ты (твоя личность)
5. Обработай входящие запросы, обновляй документацию, CHANGELOG, развивай AISEC.md
6. **Не читай код и DEVNOTES.md** — работай с позиции пользователя
7. Если нужно разобраться в проблеме — попроси AIDEV объяснить через INBOX.md

### Если ты новый агент:

1. Создай свой файл `AI<ROLE>.md` для личности (формат свободный, **читает только твоя роль**)
2. Определи зону ответственности
3. Добавь описание своей роли в этот файл
4. Начни использовать INBOX.md для коммуникации с другими агентами

## 🌐 Языковая политика

### Основной принцип

**Разработка ведётся на русском языке.** Все исходные документы создаются на русском, затем переводятся на английский для международной аудитории.

### Структура документации

| Файл | Язык | Назначение |
|------|------|------------|
| **README.ru.md** | 🇷🇺 Русский | Оригинальная витрина проекта (источник правды) |
| **README.md** | 🇬🇧 Английский | Перевод витрины для международной аудитории |
| **DOCS.ru.md** | 🇷🇺 Русский | Оригинальная полная документация (источник правды) |
| **DOCS.md** | 🇬🇧 Английский | Перевод документации для международной аудитории |
| **SPEC.md** | 🇷🇺 Русский | Техническая спецификация (только русский) |
| **DEVNOTES.md** | 🇷🇺 Русский | Рабочий журнал AIDEV (только русский) |
| **CHANGELOG.md** | 🇷🇺 Русский | История изменений (только русский) |
| **AGENTS.md** | 🇷🇺 Русский | Координация агентов (только русский) |
| **AIDEV.md** | 🇷🇺 Русский | Личность AIDEV (только русский) |
| **AISEC.md** | 🇷🇺 Русский | Личность AISEC (только русский) |
| **INBOX.md** | 🇷🇺 Русский | Коммуникация между агентами (только русский) |

### Правила перевода

1. **Оригиналы на русском** — все новые документы создаются на русском
2. **Перевод на английский** — только публичные витрины (README.md, DOCS.md)
3. **Внутренние документы** — SPEC.md, DEVNOTES.md, CHANGELOG.md, AGENTS.md, AI*.md, INBOX.md остаются на русском
4. **Пометка в оригиналах** — в начале README.ru.md и DOCS.ru.md добавляется сноска о наличии английской версии
5. **Синхронизация** — при изменении русского оригинала, английская версия обновляется следом

### Порядок работы с документацией

**Для агентов (AIDEV и AISEC):**

1. **Всегда работай сначала с русской версией** (.ru.md файлы)
2. **После изменения русской версии** — обнови английский перевод
3. **Проверяй синхронизацию** — английская версия должна соответствовать русской

**Пример:**
```
Задача: Добавить раздел в README
->
Шаг 1: Обновить README.ru.md
Шаг 2: Обновить README.md
Шаг 3: Проверить консистентность
```

### Работа с внешними Pull Requests

**Принцип:** Внешние PR приходят на английском -> синхронизируем в русскую версию

**Порядок действий:**

1. **Получен PR на английском** (например, README.md)
2. **Проанализируй изменения** — что добавлено/изменено
3. **Переведи изменения на русский** — обнови README.ru.md
4. **Синхронизируй английскую версию** — если нужно, дополни/улучши перевод
5. **Проверь консистентность** — обе версии должны отражать одно и то же

**Пример:**
```
Внешний PR: README.md (добавлен раздел)
->
Шаг 1: Перевести раздел на русский
Шаг 2: Добавить в README.ru.md
Шаг 3: Проверить английскую версию
Шаг 4: Закоммитить обе версии
```

**Итоговая таблица workflow:**

| Сценарий | Источник | Порядок действий |
|----------|----------|------------------|
| **Внутренняя разработка** | AIDEV/AISEC | .ru.md -> .md (перевод) |
| **Внешний PR** | Community | .md -> .ru.md (перевод на русский) |

### Почему так

- **Разработка на родном языке** — быстрее мышление, точнее формулировки
- **Витрины на английском** — международная аудитория, пост на Hacker News
- **Разделение** — чёткое разграничение источника правды и перевода
- **Удобство** — можно работать с оригиналом не переключая контекст
